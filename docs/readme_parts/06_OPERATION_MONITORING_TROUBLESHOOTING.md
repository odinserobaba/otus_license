# 06. Operation, Monitoring, Troubleshooting

## Базовый операционный цикл

1. Пересобрать индекс:

```bash
./build.sh
```

2. Запустить приложение:

```bash
./run.sh
```

3. Проверить интерфейс:

- открыть `http://127.0.0.1:7860`;
- сделать 3 тестовых запроса:
  - "какие документы нужны...";
  - "как переоформить...";
  - "продление лицензии...".

## Ключевые артефакты в `processed/`

- `cleaned_docs.jsonl` — merged документы;
- `chunks.jsonl` — chunks для retrieval;
- `lexical_index.json` — индекс;
- `qa_history.jsonl` — история QA;
- `chat_logs.jsonl` — подробные логи (если включено);
- `embedding_cache.json` — кэш embeddings.

## Что мониторить регулярно

### 1. Retrieval качество

- появляются ли в top-k ожидаемые НПА;
- нет ли доминирования нерелевантных документов;
- не "проваливается" ли `license.txt` в транспортных вопросах.

### 2. Генерация

- есть ли конкретные нормы в ответе;
- не смешиваются ли "документы" и "требования";
- нет ли ссылок на НПА, отсутствующие в контексте.

### 3. Техническое состояние

- время ответа;
- доступность API для Yandex;
- объем `embedding_cache.json` и скорость роста.

## Частые проблемы и решения

### Проблема: ответ слишком общий

Проверить:

- включен ли `LLM-режим`;
- достаточно ли `top_k` (обычно 6..8);
- включен ли embeddings re-rank;
- есть ли в retrieval-контексте нужные статьи.

Действия:

- включить `multi-step retrieval`;
- проверить `license.txt` секции в `processed/extra_docs.jsonl`.

### Проблема: LLM ссылается на "лишние" НПА

Проверить:

- предупреждения в блоке проверки источников;
- насколько "шумный" retrieval-контекст.

Действия:

- усилить `official_only`;
- уменьшить `top_k`;
- включить embeddings re-rank;
- использовать блок "Раскрытие норм из контекста" как приоритет.

### Проблема: embeddings не работают

Симптом:

- появляется заметка "Embeddings re-rank отключен...".

Проверить:

- API key;
- folder id;
- embedding model;
- сетевую доступность Yandex API.

### Проблема: медленная работа

Причины:

- большой `embeddings_top_n`;
- холодный кэш;
- тяжелая модель LLM.

Действия:

- снизить `embeddings_top_n` до 20..30;
- прогреть кэш типовыми запросами;
- временно отключить reasoning/multi-step.

## Режимы логирования

### Минимальный

- только `qa_history.jsonl`.

Подходит для обычной эксплуатации.

### Диагностический

- включить checkbox "Логирование в файл";
- анализировать `chat_logs.jsonl`.

Подходит для отладки retrieval и prompt-поведения.

## Полезные команды для поддержки

Проверка синтаксиса:

```bash
.venv/bin/python -m py_compile app.py scripts/*.py
```

Перегенерация датасета LoRA:

```bash
python3 scripts/build_lora_dataset.py \
  --input processed/qa_history.jsonl \
  --out-train data/lora/train.jsonl \
  --out-eval data/lora/eval.jsonl
```

## Обновление корпуса (операционный протокол)

1. добавить файлы в `doc/`;  
2. запустить `./build.sh`;  
3. smoke-test retrieval;  
4. проверить 3-5 вопросов в UI;  
5. зафиксировать изменения.

## Release-чеклист перед демонстрацией

- сборка проходит с нуля;
- UI поднимается;
- 10 целевых вопросов отрабатывают без критичных галлюцинаций;
- источники цитируются по метаданным;
- debug-технические блоки не видны пользователю.

# 03. App and RAG Pipeline

## Назначение `app.py`

`app.py` — центральный рантайм-модуль, который:

- принимает вопрос из UI;
- запускает retrieval;
- (опционально) делает re-rank embeddings;
- (опционально) вызывает LLM;
- структурирует ответ и источники;
- валидирует результат;
- пишет логи.

## Основной runtime-поток

1. Нормализация запроса и определение intent/tags.
2. Lexical scoring (`score_query`).
3. Выбор top-кандидатов (`select_diverse_matches`).
4. Опционально:
   - multi-step follow-up retrieval,
   - deterministic anchors,
   - direct article pull.
5. Сбор prompt-контекста.
6. Генерация (LLM или шаблон).
7. Пост-обработка:
   - норм-диджест,
   - список документов из контекста,
   - разделение "документы vs требования",
   - dedupe источников,
   - проверка галлюцинаций.
8. Возврат ответа + дисклеймер.

## Retrieval-логика в деталях

### Базовый скоринг

`score_query` учитывает:

- TF-IDF вклад терминов;
- entity hits (номера НПА, органы);
- intent matching (`issuance/reissue/extension`);
- topic tags (`fee`, `epgu`, `lab`, ...);
- query-adaptive boosts:
  - документные запросы,
  - транспортные запросы.

### Фильтрация `official_only`

При включенном флаге:

- неофициальные источники отбрасываются,
- но `source_kind=guide` допускается как вспомогательный слой.

## Формирование ответа

### Режимы

- `use_llm=true`: генерируется через backend (Ollama/Yandex).
- `use_llm=false`: шаблонный ответ `template_legal_answer`.

### Обязательные блоки

Система стремится к стабильной структуре:

- `Краткий ответ`
- `Нормативное основание`
- `Практические шаги`
- `Что нужно уточнить у заявителя`
- `Источники`

Если LLM пропускает блок "Что нужно уточнить у заявителя", он добавляется программно.

### Спец-блок для транспортной лицензии

Добавлен блок:

- `Разделение: документы для подачи vs требования к лицензиату`.

Цель: не смешивать "что приложить к заявлению" и "каким условиям должен соответствовать перевозчик".

## Анти-галлюцинационные механизмы

1. Строгий prompt ("только контекст", "не выдумывать реквизиты").
2. Проверка цитируемых номеров НПА относительно retrieval-контекста.
3. Удаление шумовых markdown-ссылок `](29)`.
4. Дедупликация секций источников.
5. Авто-добавление "раскрытия норм" по найденным фрагментам.

## Работа с источниками и цитированием

Вывод источников идет через `doc_label(meta)` и `doc_citation`, а не через filename.

Плюсы:

- читабельные ссылки;
- единый стиль цитирования;
- меньше "технического" мусора в ответе.

## Логи

### `processed/qa_history.jsonl`

Всегда сохраняется упрощенная история:

- `question`
- `answer`
- мета-поля исполнения.

### `processed/chat_logs.jsonl`

Заполняется при включенном логировании:

- prompt,
- preview ответа,
- диагностика retrieval/LLM.

Примечание: служебный debug-блок в UI больше не выводится пользователю.

## UI-контролы, влияющие на качество

- `Top-K`: баланс полноты и шума.
- `Только официальные НПА`.
- `LLM backend/model`.
- `Embeddings re-rank` + `top-N`.
- `Многошаговый retrieval`.
- `Показывать рассуждение модели`.

## Практические пресеты

### Пресет "стабильный"

- `top_k=6`
- `official_only=true`
- `use_embeddings_rerank=false`
- `multi_step=false`

### Пресет "усиленный"

- `top_k=6..8`
- `official_only=true`
- `use_embeddings_rerank=true`
- `embeddings_top_n=30..50`
- `multi_step=true`

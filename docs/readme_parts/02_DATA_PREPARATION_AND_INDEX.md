# 02. Data Preparation and Index

## Назначение

Этот раздел описывает полный pipeline подготовки данных:

1. нормализация исходных файлов;
2. извлечение текста;
3. чистка;
4. разметка метаданными;
5. чанкинг;
6. построение индекса.

Все шаги выполняются через `build.sh`.

## Полный pipeline `build.sh`

Скрипт выполняет:

1) создание/проверка `.venv`;  
2) установка зависимостей из `requirements.txt`;  
3) переименование файлов (`scripts/rename_all_docs.py`);  
4) подготовка RTF-корпуса (`scripts/prepare_corpus.py`);  
5) подготовка "дополнительного" корпуса (`scripts/prepare_doc_files.py`);  
6) merge в `processed/cleaned_docs.jsonl`;  
7) чанкинг и индекс.

Текущие параметры чанков задаются в `build.sh`:

- `CHUNK_SIZE=5000`
- `CHUNK_OVERLAP=1000`

Эти значения стоит периодически валидировать по качеству retrieval.

## Шаг 1: Переименование и нормализация имен файлов

Скрипт `scripts/rename_all_docs.py`:

- убирает проблемные символы;
- приводит имена к более стабильному формату;
- сохраняет соответствия в map-файл.

Зачем:

- исключить проблемы с кодировками/пробелами;
- стабилизировать source-идентификаторы в метаданных.

## Шаг 2: Подготовка RTF/MHTML

Скрипт `scripts/prepare_corpus.py`:

- извлекает полезный текст из RTF, включая MHTML-фрагменты;
- очищает "шум";
- пытается извлечь реквизиты документа;
- записывает в `processed/cleaned_docs_rtf.jsonl`.

Ключевые поля на выходе:

- `id`,
- `metadata` (тип, номер, дата, title, citation),
- `text`.

## Шаг 3: Подготовка DOC/DOCX/TXT/MD/PDF

Скрипт `scripts/prepare_doc_files.py`:

- DOCX: чтение через zip/XML;
- DOC: `antiword`/`catdoc`/`libreoffice` fallback;
- PDF: извлечение по страницам + логическое деление;
- TXT/MD: текстовое чтение + секционирование.

### Спец-обработка `license.txt`

Встроен отдельный splitter:

- не режет на каждом нумерованном подпункте;
- режет по высокоуровневым заголовкам ("Для получения лицензии...", "Прием заявлений...");
- ставит `source_kind=guide`.

Это важно для запросов "какие документы нужны", где нужна цельная секция, а не обрывки.

## Шаг 4: Merge корпусов

В `build.sh` делается объединение:

- `processed/cleaned_docs_rtf.jsonl`
- `processed/extra_docs.jsonl`

Итог: `processed/cleaned_docs.jsonl`.

## Шаг 5: Чанкинг

Скрипт `scripts/chunk_corpus.py`:

- режет документы на chunks;
- учитывает overlap;
- использует режимы:
  - обычный paragraph-aware,
  - article-aware для федеральных законов.

### Что делает article-aware

- находит границы "Глава", "Статья";
- сохраняет `article_number`, `article_title`;
- извлекает `subpoint_refs`;
- извлекает `cited_article_refs`.

Это критично для юридического retrieval по конкретным статьям и подпунктам.

## Шаг 6: Индекс

Скрипт `scripts/build_index.py`:

- строит TF-IDF словарь (`idf`);
- сохраняет токенные представления chunks;
- делает dedup похожих chunks;
- пишет `processed/lexical_index.json`.

## Проверка результата после сборки

Минимальный smoke:

```bash
python3 scripts/test_retrieval.py --index processed/lexical_index.json --top-k 6
```

Проверяем:

- индекс читается без ошибок;
- возвращаются ожидаемые источники;
- в источниках нет "сломанной" мешанины из файловых имен.

## Контроль качества данных (рекомендуемый чек-лист)

- 3-5 документов вручную проверить в `processed/clean_txt/`;
- убедиться, что даты/номера распознаны корректно;
- проверить, что для 171-ФЗ есть article metadata;
- проверить, что `license.txt` представлен секциями, а не случайными кусками;
- убедиться, что нет явного дублирования chunks.

## Типовые проблемы и быстрые фиксы

- пустые DOC: установить `antiword` или `catdoc`;
- нечитабельные PDF: проверить извлечение страницы и fallback на full text;
- чрезмерно длинные чанки: снизить `CHUNK_SIZE`, например до `2200-3200`;
- потеря контекста на границе: увеличить overlap.

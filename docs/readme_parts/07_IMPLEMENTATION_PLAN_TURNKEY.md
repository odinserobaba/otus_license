# 07. Turnkey Implementation Plan

## Цель этапа "под ключ"

Получить версию проекта, которая:

- стабильно отвечает по вашему предметному домену;
- демонстрирует улучшение качества через embeddings и LoRA;
- готова к защите (прозрачная архитектура + воспроизводимость).

## Итоговые deliverables

К моменту завершения должны быть:

1. Рабочий локальный веб-ассистент с гибридным retrieval.
2. Набор тестовых вопросов и результатов A/B.
3. LoRA-адаптер (обученный в Colab) + локальная проверка.
4. Комплект документации и слайды для защиты.

---

## Фаза A. Стабилизация данных и retrieval (1-2 дня)

### A1. Зафиксировать corpus baseline

- проверить `doc/` состав;
- обновить индекс через `./build.sh`;
- сохранить слепок `processed/lexical_index.json`.

### A2. Retrieval smoke-set

Создать список 20-30 вопросов по 4 классам:

- получение лицензии;
- переоформление;
- продление;
- транспорт/этиловый спирт.

### A3. Проверка качества baseline

Для каждого вопроса оценить:

- релевантность top-k источников;
- полноту норм в ответе;
- наличие ложных ссылок.

Критерий завершения фазы:

- baseline повторяем и стабилен.

---

## Фаза B. Гибридный retrieval (1 день)

### B1. Включить embeddings режим

Рекомендуемый старт:

- `Embeddings re-rank=true`
- `top-N=40`
- `top_k=6`

### B2. Провести A/B сравнение

Сравнить baseline vs hybrid на одном вопроснике:

- hit@k;
- экспертная оценка качества ответа;
- средняя задержка.

### B3. Зафиксировать рабочие параметры

Выбрать "рабочий профиль" и "демо-профиль":

- рабочий: быстрее и стабильнее;
- демо: максимум качества.

Критерий завершения фазы:

- есть измеримое улучшение retrieval/ответа без критичного роста latency.

---

## Фаза C. LoRA цикл (2-3 дня)

### C1. Подготовка датасета

Команда:

```bash
python3 scripts/build_lora_dataset.py \
  --input processed/qa_history.jsonl \
  --out-train data/lora/train.jsonl \
  --out-eval data/lora/eval.jsonl \
  --eval-ratio 0.1
```

### C2. Очистка датасета вручную

- убрать слабые/ошибочные ответы;
- выровнять стиль;
- оставить максимально полезные пары.

### C3. Обучение QLoRA в Colab

- запустить `notebooks/lora_qlora_colab.ipynb`;
- сохранить adapter в Drive;
- скачать adapter артефакт.

### C4. Локальный sanity-check

```bash
python3 scripts/lora_infer_local.py \
  --base-model Qwen/Qwen2.5-1.5B-Instruct \
  --adapter-path /path/to/adapter \
  --question "какие документы нужны для перевозки этилового спирта"
```

Критерий завершения фазы:

- LoRA дает прирост по стилю/структуре без заметного роста ошибок.

---

## Фаза D. Интеграция и финализация (1 день)

### D1. Интеграционный тест

Проверить 15-20 ключевых вопросов:

- с embeddings on/off;
- с LoRA baseline comparison;
- с `official_only=true`.

### D2. Операционные сценарии

- rebuild с нуля;
- запуск приложения;
- быстрая диагностика типовых сбоев.

### D3. Подготовка презентации

Обязательно показать:

1) архитектуру "дешевый baseline + усилители",  
2) A/B quality gain,  
3) антипаттерны и как вы их закрыли (галлюцинации, шум источников),  
4) практическую применимость в вашем домене.

---

## Технический Definition of Done

Считать проект "под ключ" готовым, если:

- `./build.sh` и `./run.sh` стабильно работают;
- ответы содержат проверяемые нормативные основания;
- нет утечек служебных блоков в UI;
- A/B результаты документированы;
- LoRA pipeline воспроизводим от датасета до инференса.

## Риски и их смягчение

1. Шум в QA-датасете -> ручная чистка и eval holdout.  
2. API нестабильность embeddings -> fallback TF-IDF + кэш.  
3. Слишком большие чанки -> настройка `CHUNK_SIZE/OVERLAP` по quality sweep.  
4. Галлюцинации -> строгий prompt + валидация + source checks.

## Что делать сразу после этого плана

1. Запустить A/B retrieval прогон на вашем вопроснике.  
2. Дочистить датасет и сделать первый QLoRA run.  
3. Зафиксировать результаты в отдельном отчете для защиты.

# Подробный разбор пайплайна документов и чанков

Этот документ описывает фактическую механику проекта: как из папки `doc/` формируется рабочий корпус, как строятся чанки для retrieval, и как это влияет на итоговые ответы в `app.py`.

---

## 1) Общая идея пайплайна

Пайплайн разделен на offline и online части.

- **Offline (подготовка данных):**
  1. Нормализация имен исходных документов.
  2. Извлечение/очистка текста из разных форматов.
  3. Сбор метаданных (тип НПА, номер, дата, заголовки, теги).
  4. Дедупликация и объединение в единый корпус.
  5. Чанкинг юридических текстов с сохранением структуры норм.
  6. Сборка lexical-индекса.

- **Online (во время вопроса пользователя):**
  1. Поиск релевантных чанков по индексу.
  2. (Опционально) rerank embeddings.
  3. Multi-step и иерархическое расширение контекста.
  4. Генерация ответа и пост-проверки (guardrails, sanitation, sources).

Ключевой принцип: **не просто нарезать текст на куски, а сохранить юридическую структуру**, чтобы ответ ссылался на корректные фрагменты норм.

---

## 2) Шаг 0: Нормализация имен файлов (`scripts/rename_all_docs.py`)

Перед основной обработкой `build.sh` запускает `scripts/rename_all_docs.py`.

Что делает скрипт:

- проходит по файлам `doc/*.rtf|.doc|.docx`;
- извлекает текст для попытки распознать:
  - номер документа;
  - дату документа;
- формирует стабильные имена:
  - `norm_<number>_<dd_mm_yyyy>.ext`
  - либо `norm_unknown_...` при отсутствии реквизитов;
- пишет маппинг старых/новых имен в `doc/rename_map_latest.tsv`.

Зачем это важно:

- стабильные имена уменьшают хаос в корпусе;
- номер/дата из имени потом помогают в метаданных и dedupe.

---

## 3) Шаг 1: Обработка RTF/MHTML (`scripts/prepare_corpus.py`)

Скрипт берет `doc/**/*.rtf` и формирует:

- `processed/clean_txt/*.txt` (очищенный текст каждого документа),
- `processed/cleaned_docs_rtf.jsonl` (записи для RAG).

### 3.1 Извлечение текста

У многих `.rtf` внутри MHTML/HTML. Скрипт:

1. читает байты файла;
2. пытается распарсить MIME-структуру;
3. ищет `text/html`;
4. декодирует (по умолчанию `windows-1251`);
5. удаляет HTML-мусор:
   - `<script>`, `<style>`, комментарии;
   - теги;
   - лишние пробелы/переводы строк.

### 3.2 Извлечение метаданных

Скрипт собирает поля:

- `source_file`, `source_rel_path`, `source_bucket`;
- `doc_type` (например, ПРИКАЗ/ПОСТАНОВЛЕНИЕ/ФЕДЕРАЛЬНЫЙ ЗАКОН);
- `doc_number_file`, `doc_number_text`;
- `doc_date_file`, `doc_date_text`, `doc_date_effective`;
- `title_guess`, `doc_title`;
- `doc_citation` (готовая строка цитирования).

Логика даты:

- приоритет: дата из имени файла (`doc_date_file`), если есть;
- fallback: дата из заголовка текста;
- итог: `doc_date_effective`.

Выходной JSONL-объект:

```json
{
  "id": "norm_171fz_22_11_1995",
  "metadata": { "...": "..." },
  "text": "Очищенный текст документа..."
}
```

## 4) Шаг 2: Обработка дополнительных форматов (`scripts/prepare_doc_files.py`)

Скрипт обрабатывает:

- `.doc`, `.docx`, `.txt`, `.md`, `.pdf`

и пишет:

- `processed/extra_docs.jsonl`,
- тексты секций в `processed/clean_txt/*.txt`.

### 4.1 Как читаются форматы

- **DOCX**: чтение `word/document.xml` из zip.
- **DOC**:
  - `antiword` (если установлен),
  - иначе `catdoc`,
  - иначе `libreoffice --headless` конверт в txt.
- **TXT/MD**: чтение с пробами кодировок (`utf-8`, `cp1251`, `windows-1251`).
- **PDF**: page-by-page extraction (`pypdf`), попытка разбиения по заголовкам.

### 4.2 Почему здесь есть секции

Для длинных гайдовых документов один файл может стать несколькими записями:

- разделение по верхнеуровневым заголовкам (`1. ...`, `2. ...`);
- специальный разбор `license.txt` (чтобы не ломать списки требований);
- для PDF — разделение по "Глава/Статья/Раздел", либо fallback в один блок.

Каждая секция становится отдельной записью:

- `id`: `<stem>_extra_s001`, `<stem>_extra_s002`, ...
- `section_title`, `section_index`,
- для PDF: `section_page_start`, `section_page_end`.

### 4.3 Дополнительные семантические поля

Скрипт добавляет:

- `source_kind`: `official` / `guide` / `other`;
- `procedure_type`: `issuance` / `reissue` / `extension` (по текстовым маркерам);
- `topic_tags`: например `epgu`, `fee`, `lab`, `retail`.

Это нужно для более умного retrieval и более точного форматирования ответа.

## 5) Шаг 3: Слияние и дедупликация (`scripts/merge_corpora.py`)

Источники для объединения:

- `processed/cleaned_docs_rtf.jsonl`
- `processed/extra_docs.jsonl`
- `processed/extra_docs_egais_centerinform.jsonl` (если есть)

Выход:

- `processed/cleaned_docs.jsonl` (единый корпус),
- `processed/new_doc_presence_report.jsonl` (отчет по `new_doc`).

### 5.1 Как строится dedupe-ключ

Если есть реквизиты документа:

- `doc:<doc_type>|<doc_number>|<doc_date>`

Если реквизитов нет:

- `text:<sha1(normalized_text)>`

### 5.2 Приоритизация дублей

Если документ встречается и в `doc/`, и в `new_doc/`:

- предпочтение отдается записи из `doc/` (канонической);
- дубликат из `new_doc/` маркируется как `already_in_corpus`.

Поля статуса:

- `existing` — из базового корпуса;
- `new_unique` — действительно новый материал;
- `already_in_corpus` — дубликат существующего.

## 6) Шаг 4: Юридический чанкинг (`scripts/chunk_corpus.py`)

Это один из самых важных этапов проекта.

Вход:

- `processed/cleaned_docs.jsonl`

Выход:

- `processed/chunks.jsonl`

### 6.1 Базовое разбиение на параграфы

Сначала текст превращается в список семантических блоков:

- пустая строка -> граница;
- заголовки (`Глава`, `Статья`, `Раздел`, нумерованные пункты) выделяются как отдельные блоки;
- fallback для "грязных" OCR-текстов.

### 6.2 Спец-логика для федеральных законов

Если документ распознан как ФЗ:

- дополнительно выделяются блоки статей;
- чанки строятся внутри статьи;
- сохраняется привязка к статье/главе.

Это дает лучший контроль цитат и контекста при ссылках вида "в статье N".

### 6.3 Правила нарезки chunk-size/overlap

Скрипт учитывает:

- `chunk_size` (в `build.sh` используется `3200`);
- `overlap` (в `build.sh` используется `700`).

Особенности:

- длинные параграфы режутся по предложениям;
- **списки сохраняются целостно** (разбиение по маркерам пунктов, а не случайно по середине);
- новый чанк может принудительно начинаться на "жесткой" юридической границе (`Глава/Статья/Раздел`).

### 6.4 Метаданные каждого чанка

К каждому chunk добавляются:

- технические:
  - `chunk_index`, `chunk_chars`;
- юридические:
  - `chapter_title`, `article_number`, `article_title` (если применимо),
  - `subpoint_refs`,
  - `cited_article_refs`,
  - `norm_refs`,
  - `list_density`;
- графовые/иерархические:
  - `neighbor_prev_chunk_id`, `neighbor_next_chunk_id`,
  - `hierarchy_seq_index`, `hierarchy_seq_total`,
  - `article_part_index`, `article_part_total`, `article_key`
    (для блоков статей),
  - либо `doc_part_index`, `doc_part_total` для обычной последовательности.

`norm_refs` строятся из:

- ключа НПА (номер/тип),
- номера статьи,
- подпунктов.

Именно эти ссылки помогают retrieval точнее находить фрагменты при правовых вопросах.

## 7) Шаг 5: Построение индекса (`scripts/build_index.py`)

После чанкинга строится `processed/lexical_index.json`.

Внутри:

- словарь токенов и веса (TF-IDF-подобная логика);
- связь токен -> набор chunk_id;
- метаданные по чанкам для скоринга.

Зачем это нужно:

- быстрый первый этап retrieval без вызовов внешних API;
- базовый fallback-режим, когда embeddings/LLM временно недоступны.

---

## 8) Как это используется при ответе пользователя (`app.py`)

Упрощенный runtime-поток:

1. Вопрос пользователя нормализуется.
2. По lexical-индексу находятся кандидаты.
3. (Опционально) кандидаты переупорядочиваются embeddings rerank.
4. (Опционально) multi-step строит дополнительные подзапросы и добавляет релевантные чанки.
5. Идет hierarchy expansion:
   - соседние chunk,
   - части той же статьи,
   - связанный контекст.
6. По финальному контексту генерируется ответ.
7. Применяются guardrails:
   - проверки фактов,
   - очистка источников,
   - структурирование блока "Что сделать/Какие документы/Источники".

Итог: пользователь получает не "голую генерацию", а ответ, собранный из проверяемого контекста.

---

## 9) Какие файлы считаются ключевыми для этого контура

- `build.sh` — orchestrator всего offline пайплайна.
- `scripts/rename_all_docs.py` — нормализация имен источников.
- `scripts/prepare_corpus.py` — RTF/MHTML -> чистый текст + metadata.
- `scripts/prepare_doc_files.py` — DOC/DOCX/TXT/MD/PDF -> секции + metadata.
- `scripts/merge_corpora.py` — dedupe + unified corpus.
- `scripts/chunk_corpus.py` — юридический chunking + hierarchy metadata.
- `scripts/build_index.py` — lexical index.
- `app.py` — retrieval + generation + guardrails.

## 11) Почему это устойчиво в реальном использовании

- Неполадки внешней модели не ломают базовый retrieval (есть lexical fallback).
- Юридические списки не "рвутся" в чанках, поэтому шаги и требования не теряются.
- Dedup не дает дублировать одинаковые НПА из разных источников.
- Metadata (номер, дата, статья, подпункты) помогает и поиску, и корректной цитате в ответе.
- Иерархические связи chunk уменьшают риск ответа по вырванной из контекста фразе.

- Добавить quality-metrics на уровне chunk (coverage важных статей, noisy-token ratio).
